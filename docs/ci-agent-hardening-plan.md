# CI remediation agent hardening plan

Date: 2026-05-09

## Objective

Harden the current real orchestration design with deterministic identity,
strict schema validation, and safe patch application gates before any default
auto-apply behavior.

## Required architecture updates

### 1) Deterministic `root_cause_signature` generation in orchestration code

`root_cause_signature` must not be generated by an LLM response. The
orchestrator computes:

```python
signature_input = f"{category}|{trigger_file}|{trigger_function}"
root_cause_signature = sha256(signature_input.encode("utf-8")).hexdigest()
```

The orchestrator then injects the computed value into Prompt B, C, and D
payloads.

### 2) Prompt B must include grounded file context

Prompt B inputs must include exact target file contents (or bounded line slices)
for every path in `target_files`. Prompt B is not permitted to produce a diff
for files that were not supplied with source context.

### 3) JSON schema validation + fail-closed retry strategy

Each prompt stage (A/B/C/D) must validate against an explicit JSON schema with:

- `required` fields
- strict `enum` constraints
- `additionalProperties: false`
- explicit non-null rules

When validation fails:

1. Retry once with repair instruction (`Return ONLY valid JSON matching schema exactly.`)
2. If validation still fails, terminate stage with observation-only outcome.

### 4) Safe patch application pipeline

Never commit directly from raw LLM diff output. Enforce this sequence:

1. Create temp worktree pinned to `based_on_commit`
2. `git apply --check` on generated patch
3. Apply patch
4. Syntax/static checks (`bash -n`, `shellcheck`, `shfmt`)
5. Test suite gates (`bats` and project validation target)
6. Commit only if all checks pass
7. Create PR with rollback instructions

Auto-merge remains disabled by default.

### 5) Confidence anti-hallucination constraints

Prompt A policy:

- Never output confidence `> 0.80` without `trigger_line` present in logs
- If `trigger_line` is not explicitly present in logs/traces, force confidence `<= 0.65`
- Do not infer exact line numbers from repository context

### 6) Prompt D history ordering contract

History ordering must be explicit and immutable for trend calculations.

Contract: history array order is **oldest → newest**.

Trend window definitions:

- Recent window: records `[-10:]`
- Baseline window: records `[-20:-10]`

### 7) Provenance + rollback metadata in fix proposals

Prompt B output schema must include:

- `based_on_commit` (full git SHA)
- `target_files` (non-empty list)
- `rollback_plan` containing at least one reversible command

PR body template must include a concrete revert instruction:

```bash
git revert <commit_sha>
```

## Token/input budget contract

Define and enforce orchestration limits:

- `MAX_PREPROCESSED_LOG_BYTES=16000`
- `MAX_PROMPT_TOKENS=4000`

Requests exceeding limits must be summarized deterministically in preprocessing
before model invocation.

## Minimum schema skeleton (all prompt stages)

Each schema should start from this policy baseline:

```json
{
  "type": "object",
  "additionalProperties": false,
  "required": []
}
```

Per-stage schema then extends required keys and value constraints.
